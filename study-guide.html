1The big picture

An AI model like Llama learned from huge amounts of public internet text. But it has never seen your PDF. Your file is private — it is not inside the model.

Ask it about your document anyway, and it has two bad options: admit it doesn't know, or — worse — hallucinate: invent a confident-sounding answer that is simply made up.

The fix: don't ask the model to remember your PDF. Show it the right paragraphs at question time.

Think of an open-book exam. The student (the AI) is smart but never read your book. A librarian (your code) finds the right pages and slides them across the desk. The student answers using only those pages. That system is called RAG.

Your project is that librarian + student system — and by version 3, the librarian got very good: it reads scanned pages, understands synonyms, double-checks its own work, and even retries with better search words when the first attempt looks weak.

Extract→ Chunk→ Index→ Rewrite→ Retrieve + Rerank→ Grade / Retry→ Augment→ Generate→ Self-check

Green-shadowed stages are the "smart" v3 additions. Sections 4–10 walk through every stage with examples; sections 11–12 explain the code and the full workflow.

2Ten words you need first

LLM (Large Language Model)
A program trained on massive text to predict the next word, again and again. That simple skill, at huge scale, lets it write answers and code. Yours is Llama 3.3 70B — 70 billion learned numbers (parameters) inside it, running on Groq's servers.
Prompt
Everything you send the model in one request: rules + evidence + question. The model knows only what's in the prompt (plus its old training). Nothing else.
System prompt
The first, special instruction that sets behavior — a job description. Yours says: "answer only from the excerpts, quote the evidence, cite pages, never guess."
Token
The unit models read and write — roughly ¾ of a word. Example: "unbelievable" might be 3 tokens: un·believ·able. APIs count, limit, and charge by tokens.
Context window
The maximum tokens a model can look at in one request — its whiteboard. Big, but finite. This limit is why we can't just paste whole documents in.
Hallucination & grounding
Hallucination = the model invents facts fluently. Grounding = forcing every claim to come from given source text. This whole project is a grounding machine.
Vector
A list of numbers describing something, so a computer can compare by math. Example: a fruit as [sweetness, size]: apple=[6,5], cherry=[7,2]. Texts get the same treatment — just with hundreds of numbers.
Temperature
The randomness dial for picking the next token. 0 = same careful answer every time; 1 = creative variety. Factual document Q&A wants low (yours: 0.2).
Streaming
Receiving the answer token-by-token as it's generated, instead of waiting for the whole thing. That's why answers "type themselves" in your app.
API & API key
An API is how your code talks to a program on someone else's computer (Llama on Groq's servers). The key is your password — stored in an environment variable (GROQ_API_KEY), never in code.

3What "RAG" actually means

RAG = Retrieval-Augmented Generation:

WordMeaningIn your project
RetrievalFind the few text pieces relevant to the questionHybrid search + reranking over chunks
AugmentedAdd those pieces into the promptParent sections pasted above the question
GenerationThe LLM writes the answerLlama 3.3 on Groq, streaming, with quotes

Why not paste the whole PDF into the prompt?

  • The context window is finite — a 300-page manual may not fit.
  • Irrelevant text hurts — models answer better from a small, relevant haystack.
  • Cost and speed — every token costs time (and on paid APIs, money).
Simple example

Question: "How long is the warranty?" — the model needs ~3 sentences from page 7, not all 8 pages. Retrieval's whole job is finding those 3 sentences.

4Reading the PDF rag_pipeline.py · extract_pages()

A PDF is not a text file — it's a drawing format: "place these letters at these positions." Your extractor gets text out three ways, in order:

4.1 — Normal text (pypdf)

pypdf walks each page and collects the placed letters back into plain text, one (page_number, text) pair per page. Page numbers power citations later.

4.2 — Tables (pdfplumber) v3

To a plain text extractor, a table is just words floating near each other — the rows get scrambled. pdfplumber sees the grid lines and rebuilds proper rows, which we append to the page text as cell | cell | cell lines.

Simple example

Scrambled: Weight Noise 2.8 kg 72 dB — which number belongs to which?

Rebuilt: Weight | 2.8 kg and Noise | 72 dB — unambiguous.

4.3 — OCR fallback for scanned pages (RapidOCR) v3

A scanned PDF is photographs of paper — there are no letters to extract. When a page yields almost no text, your pipeline: (1) draws the page into an image with pypdfium2, then (2) runs OCR (optical character recognition) — a small local neural network that reads letters from pixels, like your eyes do.

Real example from our tests

We generated a PDF containing only a picture of text. The OCR fallback read it back as: "scanned page test the replacement hose costs twenty dollars..." — a page that v2 would have rejected entirely.

5Chunking: small-to-big rag_pipeline.py · chunk_pages()upgraded

Whole pages are too big to search well. But here's the dilemma you can feel with the chunk-size slider: small chunks match sharply but carry too little context; big chunks carry context but match mushily. Version 3 solves it by using BOTH sizes at once:

Parent section (~360 words) — what the model READS
"The ACME X100 comes with a twelve (12) month limited warranty starting from the date of purchase. The warranty covers manufacturing defects... register within 30 days... to extend the warranty to twenty-four (24) months..."
child 1: "...twelve month limited warranty..." child 2: "...covers manufacturing defects..." ← matched! child 3: "...register within 30 days..."
  • Children (~120 words, the slider) are what we search — small enough that "this chunk matches the question" really means something.
  • Parents (~3× bigger) are what the model reads — when child 2 matches, the model receives the whole parent, so it also sees the registration detail next door.
  • Chunks pack whole sentences (never cut mid-sentence) and neighbors overlap by ~30 words, so ideas sitting on a cut line stay findable.
  • Chunks never cross pages, so every chunk keeps one page number and source filename for citations like (manual.pdf, page 7).

Contextual headers v3

A child like "It lasts twelve months." is clear to a human reading the page — but as a lonely chunk, it never mentions warranty, so warranty questions can miss it. The fix: every child is indexed with a contextual header stapled on top — like writing on the back of a photo where it was taken:

[manual.pdf, page 7] The ACME X100 comes with a twelve (12) month limited...
It lasts twelve months from the date of purchase.

The header (source + page + the parent's opening words) travels with the child into the index — so even orphan-sounding children carry their topic. This is a free, local version of a 2024 technique called contextual retrieval; the full version has an LLM write each header.

6Indexing: turning text into numbers TfidfIndex / EmbeddingIndex

Computers can't compare "meaning" directly — but they can compare vectors (lists of numbers). Your app builds two different kinds:

6.1 — TF-IDF: word-counting vectors

One number per known word. Two rules set each number:

  • TF (term frequency): a word used often in this chunk scores high here.
  • IDF (inverse document frequency): a word appearing in every chunk ("the", "and") is useless for telling chunks apart → scored near zero. A rare word ("warranty") → scored high.
Worked mini example

Three chunks: A = "the warranty lasts twelve months" · B = "rinse the filter under water" · C = "the battery charges in four hours".

The word the appears in all three → IDF ≈ 0 → ignored. The word warranty appears only in A → high IDF. So for the question "warranty length", chunk A scores clearly highest. Fast, transparent — but wash and rinse are different slots, so synonyms score zero.

6.2 — Embeddings: meaning vectors

An embedding model (all-MiniLM-L6-v2, ~90 MB, runs on your machine) is a neural network trained on billions of sentence pairs. It converts any text into a point in a 384-dimension "meaning space" where similar meanings land close together — even with zero shared words:

rinse the filter under water how do I wash it? battery runtime 45 minutes twelve month warranty meaning space (384 dimensions, drawn as 2)

6.3 — Cosine similarity: comparing two vectors

Both index types score a match the same way: cosine similarity — do two vectors point in the same direction? 1.0 = identical profile, 0 = nothing shared.

Simple example

With 2 numbers [sweetness, size]: apple=[6,5] and pear=[5,6] point almost the same way → similarity ≈ 0.98. Apple vs lemon=[1,3] point differently → low. Your app does this with 384 numbers instead of 2 — same idea.

6.4 — ChromaDB: the vector database

Embedding every chunk of a big document takes real time. A vector database stores the vectors on disk and finds nearest neighbors fast. Yours is ChromaDB, living in the .chroma folder — each unique (files + settings) combination gets a collection named by a fingerprint (hash) of the file bytes.

Real example from our tests

First index build: 0.25s of embedding work. Re-opening the same PDF: 0.00s — the fingerprint matched, so the stored vectors were reused. Industrial tools (Pinecone, FAISS, Weaviate) do this job for millions of vectors.

7Retrieval: hybrid fusion + reranking v3

7.1 — Hybrid retrieval with RRF HybridIndex · rrf_fuse()

Words and meanings fail differently: word-matching nails exact terms ("X100", "0.6 liters") but is synonym-blind; embeddings catch synonyms but can fumble rare exact codes. So the default mode runs both and merges their rankings with Reciprocal Rank Fusion: each list votes, and an item at rank r earns 1/(60+r) points.

Worked mini example

Word-ranking says: 1) Warranty 2) Battery 3) Specs. Meaning-ranking says: 1) Specs 2) Warranty 3) Filter.

Warranty: 1/61 + 1/62 = 0.0325 ← ranked well by BOTH lists → wins
Specs: 1/63 + 1/61 = 0.0323 · Battery: 1/62 = 0.0161 · Filter: 1/63 = 0.0159

Note RRF only uses positions, never raw scores — so it can fuse TF-IDF scores and cosine scores even though they live on completely different scales.

7.2 — The cross-encoder reranker Reranker

Everything so far is a bi-encoder: the question and the chunk are converted to vectors separately, then compared. A cross-encoder reads question + chunk together, in one pass, and outputs a single relevance score — far more precise, but too slow to run on every chunk.

Analogy

Bi-encoder = reading two people's diaries in separate rooms, then guessing if they'd get along. Cross-encoder = watching them have an actual conversation. Slower, much more accurate.

So the modern pattern your app uses: retrieve ~16 candidates cheaply → rerank them with the cross-encoder → keep the best k. The reranker is another small local model (ms-marco-MiniLM, ~80 MB).

Real example from our tests

For "How long is the warranty?" the reranker gave the warranty chunk confidence 0.98; for a totally unrelated question, its best score was 0.00. And in the full measurement, reranking pushed the correct page to rank #1 on all 8 test questions.

9The agentic loop: a pipeline that makes decisions advanced_rag.py · smart_retrieve()v3

Classic RAG is a fixed conveyor belt: retrieve → answer, no matter how bad the evidence is. Version 3 gives the pipeline judgment. Four small "thinking" calls power it (all in chatbot.py):

9.1 — Query rewriting (follow-ups)

A follow-up like "and what does it cover?" retrieves terribly — those five words say nothing about warranties. Before searching, one quick model call rewrites it into a standalone question.

Real example from our tests

You typed: and what does it cover?
Searched instead: What does the twelve-month warranty, as mentioned on page 7, cover?
(The model still receives your original words — the rewrite is only for searching.)

9.2 — Grading the evidence (CRAG)

After retrieving, the pipeline asks the model one tiny question: "Can the question be fully answered using ONLY these excerpts? YES or NO." If NO, it doesn't give up — it retries.

9.3 — Multi-query expansion (the retry)

The retry generates 2 rephrasings of the question using different words, searches with all of them, and fuses everything with RRF. Different words → different chunks get a chance.

Real trace from our tests (off-topic question) 
Q: "What is the airspeed velocity of an unladen swallow?"
- Reranked 7 candidates (best score 0.00)
- Grade: NOT sufficient - trying synonym queries (CRAG retry)
- Query variants: "What is the wind speed of a swallow in flight..." /
                  "At what rate does a burden-free swallow travel..."
- Attempt 2: fused pool now 8 candidates (still ~0.00)
- Final evidence: weak -> the answer honestly says the document doesn't cover it

The loop tried harder, then told the truth — that's agentic RAG in one trace.

9.4 — HyDE: searching with a fake answer

HyDE (Hypothetical Document Embeddings) is a clever trick: questions and answers are shaped differently, and your chunks are answer-shaped. So the model writes a short fake ideal answer (never shown to you), and the search uses THAT as the probe — an answer-shaped probe lands closer to answer-shaped chunks in meaning space.

Simple example

Question probe: "How do I wash it?" · HyDE probe: "To clean the appliance, rinse the filter under cold water and let it dry..." — the fake answer naturally uses manual-style words like rinse, which match the real page directly. In our test, HyDE found the rinse page for the wash question. It's a sidebar toggle (off by default).

Every decision — rewrite, rerank scores, grade verdict, retry, final evidence — is written into the Agentic trace you see in the expander. Nothing is hidden.

10Answering, evidence quotes, and the self-check chatbot.py

10.1 — Augment: building the prompt

# Layer 1 - the system prompt (the rules, every request):
"Answer using ONLY those excerpts... say the document doesn't cover it...
 cite (manual.pdf, page 3)... include the exact supporting sentence as a quote..."

# Layer 2 - the user message (fresh each question):
"""Document excerpts:
[Excerpt from manual.pdf, page 7]
...the PARENT section text...

Question: How long is the warranty?"""

This is grounding: the question travels wrapped in evidence. Note the excerpts are the parents (big sections), found via the children (small chunks) — small-to-big in action.

10.2 — Generate: Groq, streaming, fallback, memory

  • Groq runs the open Llama 3.3 70B model on fast chips with a free daily quota.
  • Streaming (stream=True) yields tokens as they're made — Python's yield hands each piece to the UI instantly.
  • Model fallback: if Groq ever retires the model name, the code automatically tries the next one in its MODELS list (we tested this with a fake dead model).
  • The API is stateless — the model remembers nothing between requests. The code re-sends recent history each time, trimmed to the last 6 question/answer pairs so long chats stay inside free-tier token limits.
  • Evidence quotes: the rules ask for the exact supporting sentence as a quote — so answers show their proof inline: "A full charge takes approximately four hours." (page 3)

10.3 — The groundedness self-check v3

After the answer streams, one more call plays teacher: "check every claim in this answer against these excerpts — reply ALL SUPPORTED or list the unsupported claims." The result shows under the answer as ✅ or ⚠️.

Real example from our tests

We fed the checker a fake answer: "The warranty lasts fifty years and includes free helicopter delivery (page 7)." It flagged both lies: * The warranty lasts fifty years · * includes free helicopter delivery. This is hallucination detection — a second AI pass guarding the first.

Three ideas that make the app tick

  • Streamlit reruns the whole script on every click. Anything worth keeping — the index, the bot, the transcript — lives in st.session_state, a box that survives reruns.
  • Heavy models load once. @st.cache_resource keeps the embedder and reranker in memory across reruns instead of reloading them per question.
  • A fingerprint detects change. The app hashes the uploaded file bytes + your indexing settings; a new fingerprint → rebuild the index and start a fresh chat. The same fingerprint names the ChromaDB collection — that's how re-uploads skip re-embedding.

Where is the AI, exactly? Three small neural networks run locally and never write a word: the embedder (text→vectors), the reranker (scores pairs), and OCR (pixels→letters). The only text-writing AI is Groq's Llama — called for: the rewrite, the grade, the expansion, HyDE (if on), the answer, and the self-check. Everything else is ordinary deterministic Python.

12The full workflow, step by step

Journey A — when you upload PDFs

  1. Fingerprint. The app hashes the file bytes + settings. Seen before? Skip to chat.
  2. Extract. Each page: pypdf text → pdfplumber tables appended → OCR if the page was image-only.
  3. Chunk. Pages → parent sections → child chunks, each child stamped with its contextual header, page, and source filename.
  4. Index. TF-IDF matrix built in memory; embeddings computed (or reused from .chroma if the fingerprint matches) — hybrid mode keeps both.
  5. Fresh start. A new chatbot and empty transcript go into session_state.

Journey B — when you ask a question

  1. Rewrite — if it's a follow-up, one model call makes it standalone ("and what does it cover?" → "What does the twelve-month warranty cover?").
  2. Probe — the search text is the rewritten question (or a HyDE fake answer, if toggled on).
  3. Retrieve — the index returns ~16 candidate children; in hybrid mode, both rankings are fused with RRF.
  4. Rerank — the cross-encoder re-scores question+chunk together and keeps the best.
  5. Grade — "is this enough to answer?" If NO: expand into synonym queries, retrieve again, fuse, rerank (CRAG retry).
  6. Small-to-big hand-off — the winning children are swapped for their unique PARENT sections, capped at k.
  7. Show your work — the Agentic trace + parent excerpts render in the expander, so you see exactly what the model will see.
  8. Augment + Generate — rules + parents + your original question + the last 6 Q&A pairs go to Groq; the answer streams in with an evidence quote and citations.
  9. Self-check — every claim is verified against the excerpts; ✅ or ⚠️ appears, plus the Sources line (file + pages).
  10. Remember — the turn is saved to the bot's history (for the model) and the transcript (for your screen).

Worst case that's ~4–5 model calls per question — which is why the sidebar lets you switch the advanced steps off when you want speed or are near the free-tier rate limit.

13Measuring quality — never guess eval_retrieval.py · eval_answers.py

How do you know an upgrade helped? You measure. Your project has two examiners:

13.1 — Retrieval hit rate

eval_retrieval.py asks questions where the right page is known, and reports how often that page lands in the top k — for all four configurations. Real results on the sample manual:

ConfigurationHit rate (top-4)Correct page ranked #1
TF-IDF (words)6/84/8
Embeddings (meaning)8/88/8
Hybrid RRF8/86/8
Hybrid + reranker8/88/8

Honest reading: on this small clean manual, embeddings alone already excel — hybrid + reranking earn their keep on bigger, messier documents full of exact codes and part numbers. The point is: now you can check instead of assume.

13.2 — LLM-as-judge answer scoring

eval_answers.py runs the whole pipeline per question, then a judge model grades each answer 1–5 on faithfulness (only from excerpts?) and relevance (actually answers the question?). A second AI grading the first — the same "LLM-as-judge" pattern used in industry (RAGAS).

Real example from our tests

3 questions → faithfulness 5.0/5, relevance 4.7/5. The judge even wrote sensible comments, e.g. for "How do I wash it?": "mostly relevant but does not fully address washing the entire vacuum" — a fair 4/5.

14Why it still fails sometimes

  • OCR is basic. Clean scans work; handwriting and blurry photos don't. First OCR use downloads its models (~15 MB).
  • Tables are flattened to text rows — fine for lookups, weak for giant merged-cell tables.
  • Retrieval can still miss. If the needed fact hides in a low-ranked parent, the model never sees it. Raise k, adjust chunk size — and measure.
  • The self-check is not perfect. It catches unsupported claims well, but it can only judge against what was retrieved — it can't know about pages that were never found.
  • Free-tier rate limits. All toggles on ≈ 4–5 calls per question; turn some off if Groq asks you to slow down.
  • "Summarize everything" questions don't fit top-k retrieval — that's what GraphRAG (a project-4 idea) is for.

16Check yourself

Why do we search small child chunks but give the model big parent sections?

Small pieces make similarity scores meaningful (sharp matching); big pieces give the model enough surrounding context to answer well. Small-to-big uses each size for what it's good at.

Why does "wash" find "rinse" in embeddings mode but not TF-IDF mode?

TF-IDF has one slot per exact word — "wash" and "rinse" are different slots, overlap zero. The embedding model learned from billions of sentences that they appear in the same contexts, so it places them close together in meaning space.

Why does RRF fuse RANKS instead of raw scores?

TF-IDF scores and cosine scores live on different scales — adding them directly would let one list bully the other. Positions (1st, 2nd, 3rd...) are scale-free, so each list gets an equal vote.

Why is the cross-encoder more accurate than the embedding search — and why not use it for everything?

It reads question + chunk together, so it can spot the actual connection instead of comparing two separately-made summaries. But it must run once per pair, which is far too slow for thousands of chunks — so we retrieve ~16 cheaply first, then rerank only those.

What does the "grade" step change compared to classic RAG?

Classic RAG always answers from whatever was retrieved, good or bad. The grade step adds judgment: weak evidence triggers a retry with synonym queries before answering — the pipeline becomes a loop with decisions (agentic RAG).

Why does searching with a FAKE answer (HyDE) work?

Chunks are answer-shaped; questions aren't. A model-written fake answer uses the same style and vocabulary as real document text, so it lands closer to the right chunks in meaning space.

What can the groundedness self-check catch — and what can it never catch?

It catches claims not supported by the retrieved excerpts (fabrications, wrong numbers). It can NOT catch what retrieval never found — if the right page wasn't retrieved, the checker has no way to know it exists. Retrieval quality still comes first.

The model answered a follow-up correctly. How, if the API is stateless?

The code re-sends the recent conversation (last 6 Q&A pairs) with every request. The memory lives in your Python program, not in the model.

Where exactly does text-writing AI run in this project?

Only in the Groq calls: rewrite, grade, expansion, HyDE (if on), the answer, and the self-check. The embedder, reranker, and OCR are neural networks too — but they run locally and only convert or score; they never write a word. Everything else is deterministic Python.
